SmartCAMPOST Conceptual Data Model (MCD) Documentation
1. Introduction
The SmartCAMPOST system digitalizes parcel operations at CAMPOST: creation, pickup, routing, tracking, delivery, notifications, and payments. This document describes the conceptual data model (MCD) with entities, attributes (and indicative data types), and relationships.

3. Relationships & Cardinalities
Relationship
Cardinality
Meaning
CLIENT → PARCEL
1 → 0..*
One client can send many parcels.
CLIENT → ADDRESS
1 → 0..*
One client can store multiple addresses.
PARCEL → ADDRESS (SenderAddress)
1 → 1
Each parcel has exactly one sender address.
PARCEL → ADDRESS (RecipientAddress)
1 → 1
Each parcel has exactly one recipient address.
PARCEL → PICKUP_REQUEST
1 → 0..1
A parcel may have at most one pickup request.
PARCEL → SCAN_EVENT
1 → 0..*
A parcel generates multiple scan events.
SCAN_EVENT → AGENCY
0..1 → 0..*
Each scan may occur at one agency/hub.
SCAN_EVENT → AGENT
0..1 → 0..*
Each scan is performed by one agent.
PARCEL → DELIVERY_PROOF
1 → 0..1
A parcel may have one final proof of delivery.
PARCEL → PAYMENT
1 → 0..*
A parcel can have several payments.
PAYMENT → INVOICE
1 → 0..1
A payment may issue one invoice.
PARCEL → NOTIFICATION
1 → 0..*
A parcel triggers many notifications.
CLIENT → NOTIFICATION
1 → 0..*
A client receives many notifications.
TARIFF → PRICING_DETAIL
1 → 0..*
One tariff can be applied many times.
PARCEL → PRICING_DETAIL
1 → 0..*
A parcel can have one or more applied tariff records.
Notes
Two distinct 1:1 relations from PARCEL to ADDRESS must be labeled with roles: SenderAddress and RecipientAddress.
If you require multiple proofs (e.g., photo + signature), change PARCEL → DELIVERY_PROOF to 1 → 0..* and keep ProofType as the discriminator.
PRICING_DETAIL plays the associative (and auditable) role between PARCEL and TARIFF.
